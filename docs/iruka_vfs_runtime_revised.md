# Harness Engineering、Runtime 与 Iruka_VFS

## 1. 什么是 Harness Engineering

2026 年 2 月，**Harness Engineering** 在 AI 工程圈迅速升温。OpenAI 发布《Harness engineering: leveraging Codex in an agent-first world》后，这个术语开始进入越来越多关于 Agent 系统的讨论。

所谓 Harness Engineering，可以理解为：

> **围绕模型搭建的一整套执行与约束系统。**

它关注的不是“模型会不会生成”，而是这些更现实的问题：

- Agent 在哪里执行行为
- 它能操作什么，不能操作什么
- 状态如何保存和延续
- 结果如何反馈回来
- 失败后如何恢复
- 宿主系统如何与 Agent 交接控制权

所以，模型更像“脑”，而 Harness 更像“身体 + 工作台 + 约束系统 + 执行环境”。

这点很重要。很多 Agent 系统虽然也在谈 Harness，但落地时，重点仍停留在：

- 多接几个 tools
- 多写几条 prompt 规则
- 多做几层 routing
- 多包几层 function calling schema

这些当然有价值。但当任务进入**持续编辑、持续执行、持续修正**阶段，重点就不该只放在 tools 上，而应该回到一个更基础的问题：

> **Agent 到底是在什么样的 Runtime 里做事？**

---

## 2. 为什么 Harness Engineering 的重点应该回到 Runtime

Anthropic 在做长时间运行 Agent 的过程中，总结过几类典型失败模式：

### 失败模式 1：试图一步到位（One-shotting）
Agent 倾向于一次做完所有事情，结果在实现进行到一半时上下文窗口就耗尽。下一个会话看到的是半成品、没有文档的代码，只能花大量时间猜测之前发生了什么并恢复工作状态。

### 失败模式 2：过早宣布胜利
在项目后期，当部分功能已经完成后，Agent 会环顾四周，看到已有进展就直接宣布任务完成——即使还有大量功能未实现。

### 失败模式 3：过早标记功能完成
在没有明确提示的情况下，Agent 写完代码就标记为“完成”，却没有做端到端测试。单元测试或 curl 命令通过了，不代表功能真正可用。

### 失败模式 4：环境启动困难
每次新会话启动时，Agent 需要花费大量 token 弄清楚如何运行应用、如何启动开发服务器，而不是把时间花在实际开发上。

这些问题共同指向一件事：真正的任务执行不是调用一次工具就结束，而是一个持续闭环：

- 生成行为
- 执行行为
- 观察反馈
- 根据反馈修正下一步
- 直到任务收敛

如果没有 Runtime，这条链路就只能停留在“生成建议”这一层。模型也许能输出计划、JSON、tool call 或操作步骤，但这还不是任务完成。只有这些行为真的进入某个可执行环境，被运行、被反馈、被修正时，Agent 才真正开始“工作”。

对于 **editing-style Agent** 来说，问题的本质并不是再补几个文件工具，也不是直接引入完整 sandbox，而是为 Agent 提供一个**可持续、可恢复、可约束的工作区运行时**。换句话说，这类任务的核心已经从 **tool design** 转向 **runtime design**。

## 3. 为什么文件系统对 Harness Engineering 很重要

从 Harness Engineering 的角度看，文件系统的重要性远不只是“给 Agent 多一个 read/write tool”。

它真正提供的，是一种可持续组织任务、状态与上下文的结构。Harness 要解决的，从来不只是“模型能不能做一个动作”，而是：

- 任务如何持续推进
- 上下文如何稳定存在
- 中间状态如何组织
- 不同输入和产物如何归位
- 宿主系统如何和 Agent 交接结果
- 失败后如何恢复现场

而文件系统天然适合承接这些需求。它能提供的不只是读写，而是：

- **命名空间**：把不同信息组织到不同路径下
- **层级结构**：主文件、上下文文件、辅助资料、技能文件可以自然分层
- **持续状态**：工作结果不会像 prompt 一样每轮重组
- **局部修改能力**：可以围绕某个文件、某个目录持续 patch
- **天然的宿主映射方式**：业务对象可以被投影成文件，再被回写回去

也就是说，文件系统对 Harness 的价值，不在于它是一个“工具”，而在于它是一个**工作区结构**。

对于编辑型任务，这一点尤其重要。因为这类任务通常不是一次性生成一个结果，而是围绕一组材料持续工作：

- 主文档
- 参考资料
- 临时草稿
- 技能说明
- 中间产物
- 最终输出

如果没有一套稳定的文件工作区来承接这些对象，系统就只能把它们拆散到 prompt、tool 参数、session state、数据库字段和临时缓存里，然后每一轮都重新拼装现场。

这其实不是在“做 Agent”，而是在反复重建任务现场。

## 4. 为什么我会设计 Iruka_VFS：从文件系统价值到 Workspace Runtime

如果顺着上一章继续往下推，一个很自然的问题就会出现：

既然文件系统对 Harness Engineering 的真正价值，不在于提供 read/write 这样的单点能力，而在于它能为 Agent 提供一个可持续组织任务、状态与上下文的工作区，那么工程上真正应该设计的，到底是什么？

答案并不是继续往 toolset 上加几个文件操作接口，也不是直接把问题推给 Docker、shell 或完整 sandbox。真正需要补上的，是一层专门面向编辑型任务的 **Workspace Runtime**。

**Iruka_VFS（Virtual File System）** 就是在这个判断下被设计出来的。

它要解决的问题，不是“如何让 Agent 能操作文件”，而是：

- 如何让 Agent 围绕一组文件持续工作，而不是每轮重新拼装现场
- 如何让上下文、中间产物、主文档、参考材料有稳定归位方式
- 如何让编辑过程具备连续性，而不是一串离散的 tool call
- 如何让宿主系统把业务对象自然投影进工作区，再从工作区取回结果
- 如何让失败恢复、断点续做、结果接管变得简单而自然

换句话说，Iruka_VFS 服务的不是“文件读写”本身，而是“围绕文件持续做事”这件事。

### 4.1 它不是在提供“更多文件工具”

如果只是从 tool 的角度理解文件系统，那么很容易把系统设计成：

- read_file
- write_file
- list_dir
- search_file
- patch_file

这样做并不是没用，但这些能力仍然是离散的。Agent 每次调用它们，都像是在一堆散落接口里临时拿工具办事。工具有了，但“工作现场”并没有真正建立起来。

而编辑型任务真正需要的，是一个统一、稳定、可持续推进的文件环境：

- 主文档始终在这里
- 参考材料始终在这里
- 技能说明始终在这里
- 中间草稿始终在这里
- 最终产物也始终在这里

这样一来，编辑不再是一次次瞬时操作，而是在同一个工作区里连续发生。

### 4.2 它不是在模拟完整系统

另一条常见路径，是既然任务需要持续环境，那就直接给 Agent 一个完整 sandbox：

- 完整 shell
- 真实进程
- 依赖安装
- 容器挂载
- 操作系统级隔离
- 可执行脚本与测试命令

这套东西在 coding agent、自动化执行、测试验证场景里非常重要。但对大量编辑型任务来说，它往往又太重了。

因为这类任务真正依赖的，通常不是完整 OS 语义、进程管理、包管理器或长链路命令执行；它们真正需要的是：

- 一组文件能稳定存在
- 这些文件能被持续组织
- 某个文件能被局部修改
- 上下文材料能持续被引用
- 中间产物能自然沉淀下来
- 宿主系统能随时接管结果

所以 Iruka_VFS 并不追求成为一个完整 sandbox。它不试图模拟整个系统世界，而是更克制地服务于文件型任务真正需要的那部分连续环境。

sandbox 的另一个问题是，环境初始化成本高，状态保存成本也高。为了保留工作结果、追踪变更和支持恢复，往往还需要额外搭配 git 或其他持久化机制。一旦生产中某个 sandbox 被意外释放，之前的工作区状态和中间现场就可能很难完整恢复。这会明显增加基建与运维的复杂度。

### 4.3 它补的是 Toolset 与 Sandbox 之间的空档

从工程结构上看，很多 Agent 系统其实卡在两个极端之间：

一端是 **纯 toolset**。能力很轻，但过于碎片化。Agent 能做动作，却很难在一个稳定现场里持续推进任务。

另一端是 **完整 sandbox / container runtime**。能力很强，但对很多编辑型任务来说过重，成本更高，控制面也更复杂。

问题在于，这两者之间长期缺了一层：

- 比 toolset 更完整
- 比 container runtime 更轻
- 比真实 shell 更可控
- 比纯数据库字段编辑更自然
- 更适合承接“围绕文件持续工作的任务”

Iruka_VFS 补的，正是这一层空档。它不是要替代 sandbox，而是要覆盖那类“还不需要完整系统，但已经明显超过离散工具能力”的任务区间。

这类任务非常常见，例如：

- 长文编辑
- 文档重写
- 多材料归并
- 章节式创作
- 参考资料驱动生成
- 多阶段草稿推进
- Skill 辅助的内容加工
- 需要保留中间状态的持续修改任务

这些任务如果只靠 prompt + tools，会越来越碎；如果直接上 Docker，又常常过重。所以它们需要的，不是更强的 shell，而是一个更合适的 workspace runtime。

### 4.4 Iruka_VFS 的设计定位

因此，Iruka_VFS 的定位可以更准确地表述为：

> **Iruka_VFS 不是一个单纯的文件工具集合，也不是一个完整 sandbox，它是面向 Editing Harness 的 Workspace Runtime。**

这个定位意味着它关心的重点是：

- 为 Agent 提供稳定的工作区，而不是只提供操作接口
- 为持续编辑提供现场，而不是只提供瞬时动作
- 为宿主系统提供自然的输入输出承接点，而不是只做临时读写
- 为多轮任务提供状态连续性，而不是把连续性全部压回 prompt 和 session state
- 为失败恢复和结果接管保留现场，而不是让系统每次从头对齐

## 5. Iruka_VFS 是如何实现这些能力的

从实现上看，Iruka_VFS 并不是把“文件系统”当成一个孤立模块，而是围绕一个核心目标来组织的：

> **给 Agent 提供一个可控、可持续、可缓存、可回写的虚拟工作区。**

这意味着它实现的，不只是文件读写能力，而是一整套围绕工作区展开的运行时机制，包括：

- 宿主对象到虚拟文件树的映射
- 多轮任务中的工作区初始化与复用
- 面向 Agent 的受控命令执行环境
- 面向高频编辑的运行时镜像与缓存
- 面向持久化边界的 flush / checkpoint 机制
- 面向宿主系统的结果回写通道

### 5.1 先把宿主世界投影成虚拟工作区

Iruka_VFS 的第一步，并不是让 Agent 直接操作宿主文件，而是先把宿主侧真正需要暴露给任务的对象，投影进一个统一的虚拟目录树里。

在宿主系统内部，原始对象往往是：

- 文档正文
- 章节内容
- 上下文资料
- 技能说明
- 用户草稿
- 业务附属文件

这些对象在宿主里可能来自不同模型、不同表、不同存储。如果直接把这些业务结构暴露给 Agent，Agent 看到的将是业务语义，而不是文件语义，工作方式会变得很碎。

所以 Iruka_VFS 做的第一件事，是把这些输入统一组织成一棵稳定的工作区树，例如：

```text
/workspace
  /files
  /notes
  /context
  /skills
```

在这个过程中，宿主系统并不需要把所有业务对象都“文件系统化”。它只需要提供那些确实要交给 Agent 使用的内容，然后由 VFS 把它们挂载到对应路径下。

### 5.2 用 RuntimeSeed 定义一次工作区启动所需的最小上下文

为了把宿主上下文投影成工作区，Iruka_VFS 需要一套非常克制的初始化描述。这就是它的 **RuntimeSeed** 思路。

RuntimeSeed 关注的不是宿主业务全貌，而只是一次工作区启动真正需要的最小集合，例如：

- 当前运行时身份
- 租户隔离信息
- 主文件来源
- 初始 workspace 文件
- context 文件
- skill 文件
- 扩展 metadata

这个抽象非常重要，因为它把“宿主业务的复杂性”和“Agent 工作区的可用性”切开了。也就是说，Iruka_VFS 并不要求宿主先变成一个完整文件系统；它只要求宿主回答几个更简单的问题：

- 这次任务是谁的工作区
- 哪个文件是主文件
- 哪些辅助材料要一并放进来
- 这些内容挂到什么路径下

这样一来，工作区初始化就变成了一种 **runtime seed** 问题，而不是数据库直连问题。

### 5.3 用 WritableFileSource 建立宿主与工作区之间的回写桥

仅仅把文件投影进来还不够。Editing Harness 真正需要的是：任务做完之后，结果还能自然回到宿主世界。

Iruka_VFS 对这个问题的处理很干净：它不让 Agent 直接理解宿主 ORM、业务模型或者数据库表，而是把宿主可写文件抽象成一个很窄的桥：

- `read_text()`：从宿主读取原始内容
- `write_text(text)`：把 flush 后结果写回宿主
- `virtual_path`：声明它在虚拟工作区里的挂载路径

这个设计的好处在于，宿主接入成本非常低。宿主不需要配合一整套复杂协议，只要实现“怎么读文本、怎么写文本”，Iruka_VFS 就能接管中间那层工作区运行时。

### 5.4 用 ensure 把“任务现场”真正建起来

Iruka_VFS 里很关键的一步，是 `ensure`。它的作用不是简单“检查一下目录有没有”，而是把当前工作区真正构造成一个可供 Agent 持续工作的现场。

这一步通常会完成几件事：

- 校验 workspace 身份与租户归属
- 注册当前 runtime 上下文
- 创建固定目录结构
- 注入主文件、上下文文件、技能文件、初始化文件
- 更新工作区元数据
- 创建或恢复 shell session
- 构建当前工作区的运行时镜像

也就是说，`ensure` 的意义不是“初始化一个文件夹”，而是把宿主侧的一次任务上下文，转化成一个真正可运行的工作区。

### 5.5 用 host / agent 模式切换定义控制权边界

Iruka_VFS 的一个很有价值的设计，是它没有把宿主和 Agent 看成“同时并发操作同一份状态”的关系，而是明确把工作区划分成两种阶段性模式：

- **host 模式**
- **agent 模式**

在 host 模式下，宿主可以通过 Python API 直接读写虚拟文件；在 agent 模式下，Agent 通过 `bash(...)` 在工作区里执行命令。

这样做的好处很直接：

- 明确控制权交接
- 减少并发状态混乱
- 更容易定义持久化边界

例如，Agent 一轮执行完后，宿主切回 host 模式，检查结果，再决定是否 flush。

### 5.6 用受控命令运行时，而不是真实 shell

Iruka_VFS 虽然提供了 `bash(...)` 风格入口，但它的核心并不是调用宿主系统的真实 bash，而是执行一套受控命令运行时。

如果直接把命令转发到真实 shell，就会立刻带来很多问题：

- 权限边界难控制
- 宿主文件系统暴露过多
- 命令副作用不可预测
- 审计和回放困难
- 行为很难约束到“编辑型任务需要的那部分能力”

所以 Iruka_VFS 采取的是另一条路线：实现一组足够像 shell、但仍然受控的命令，例如：

- `pwd`
- `cd`
- `ls`
- `cat`
- `rg` / `grep`
- `mkdir`
- `touch`
- `tree`
- `edit`
- `patch`

并支持有限 shell 语义，比如：

- `&&`
- `;`
- `|`
- `>`
- `>>`
- `2>&1`

这样，Agent 获得的是一个足够接近 shell 的编辑工作环境，但所有动作都仍然被限制在虚拟工作区内部。

### 5.7 把 edit / patch 作为核心写入语义

在编辑型任务里，真正高频的动作不是“整文件覆盖写入”，而是：

- 定位某段内容
- 做局部替换
- 应用补丁
- 校验上下文
- 失败时返回结构化冲突

所以 Iruka_VFS 的重点并不是单纯提供 `write_file`，而是把 `edit` 和 `patch` 作为一等公民。

这意味着系统关注的是：

- 如何做局部编辑
- 如何保留上下文一致性
- 如何让补丁失败变成可恢复反馈
- 如何让 Agent 围绕同一份材料反复推进，而不是总是整段重写

尤其是 patch 的设计，本质上服务于一个很关键的目标：**让编辑失败也变成可计算、可解释、可继续推进的中间状态。**

## 6. Iruka_VFS 如何支撑持续状态、性能与恢复

### 6.1 用 WorkspaceMirror 把数据库状态和热运行时状态分离

很多系统一开始做虚拟文件系统时，会很自然地走向一种设计：

- 文件树存在数据库里
- 每次命令执行就查库
- 每次文件修改就落库
- 每次目录遍历也查库

这样做概念上简单，但运行起来会很快遇到问题：

- 高频命令导致数据库成为热路径
- 路径解析和目录遍历开销高
- 编辑过程延迟大
- 多轮持续任务会反复命中相同状态

Iruka_VFS 的解法是把持久化状态和热执行状态分开。

数据库负责的是：

- 工作区恢复
- 持久化真值
- 审计与持久记录

而真正供运行时高频使用的，是 **WorkspaceMirror**：

- 节点镜像
- 路径索引
- 父子索引
- 当前 cwd
- workspace metadata
- revision / checkpoint_revision
- dirty 标记集合

也就是说，工作区一旦被 ensure 并加载，后续大量命令其实是在一份内存镜像上运行的。

### 6.2 先写 mirror，再按边界 checkpoint / flush

有了 mirror 之后，写路径也发生了变化。Iruka_VFS 并不要求每次编辑立即强持久化，而是采用一种更符合 Agent 编辑任务特征的策略：

- 命令执行时先写 mirror
- 标记 dirty 内容或 dirty 结构
- 推进 revision
- 在合适边界进行 checkpoint / flush

这样做有两个很大的现实意义。

第一，编辑过程会更顺。Agent 往往会在一轮里连续执行多次读写、搜索、patch、替换。如果每次都同步落库，成本很高，也很不自然。

第二，系统终于拥有了明确的 durability boundary。也就是说，不是每个动作都必须立刻 durable，而是在“这一轮告一段落”时，由宿主显式选择 flush。

与 sandbox 方案相比，这条路线不把整个执行环境都做成沉重 sandbox，而是把“要恢复的核心现场”收敛到工作区状态本身，并通过 mirror + checkpoint 管理它。这使得它在编辑型任务里更轻，也更容易定义恢复边界。

### 6.3 用 flush 定义宿主系统可接管的持久化时刻

在 Iruka_VFS 里，`flush()` 很重要，因为它不是一个普通 API，而是一个工作区结果交接点。

在这个时刻，系统会把当前工作区的脏状态安全写回持久层，包括：

- 脏节点内容
- 目录结构变化
- session 状态，例如 cwd
- workspace metadata
- 主文件回写到宿主 source

这意味着 flush 的意义，不只是“写数据库”，而是把运行时现场中已经稳定下来的结果，正式提交给宿主系统。

### 6.4 用后台 checkpoint worker 支撑异步恢复与可靠落盘

如果只有显式 flush，系统依然偏“同步式”。而真正要让这套 runtime 在长期运行里更稳，还需要后台 checkpoint 机制。

Iruka_VFS 在这部分做得很完整：

- 通过队列调度待 flush 的 workspace
- 支持 debounce，避免高频重复刷写
- 失败后支持重试
- 用指数退避降低故障放大
- 超过阈值进入 dead letter
- 保留错误 payload 便于排查

这意味着它不是把 checkpoint 当成一个临时补丁，而是真的把“失败恢复”和“后台回写”当成 runtime 能力的一部分。

### 6.5 用 Memory Cache 补足更细粒度的热点优化

除了 WorkspaceMirror 这种“整工作区热态”之外，Iruka_VFS 还做了另一层更细粒度的优化：**Memory Cache**。

它缓存的是文件内容级热点，包括：

- 按 file_id 存内容和版本
- LRU 管理
- dirty 集合
- 后台批量刷写
- 版本号乐观并发控制

从工程结构看，这形成了两层加速路径：

- **工作区级镜像**：服务连续执行场景
- **文件级缓存**：服务局部热点读写场景

这说明 Iruka_VFS 的实现思路不是“只有一种最优路径”，而是接受不同运行条件下的退化与优化组合。

### 6.6 从 benchmark 看 Iruka_VFS 的性能与整体可靠性
我做了一轮较完整的 PostgreSQL benchmark。测试参数是：
- `8` 个并发 workspace
- 每个 workspace 执行 `24` 条命令
- 每个 workspace 的文本大小 `64 KB`

在这组数据里，单工作区热态命令延迟已经明显进入“内存工作区”量级：

| 场景 | Mean | P95 | 说明 |
| --- | ---: | ---: | --- |
| `ensure_warm` | `0.31 ms` | `0.36 ms` | 热工作区复用成本很低 |
| `cat_chapter` | `1.13 ms` | `1.28 ms` | 读取主文件基本不再受数据库热路径限制 |
| `search_workspace` | `6.91 ms` | `7.16 ms` | 搜索仍然高于纯读取，但已控制在个位数毫秒 |
| `edit_chapter` | `1.65 ms` | `1.84 ms` | 高频局部编辑可以在 mirror 上快速完成 |

并发结果更能说明这套 runtime 的定位已经成立：

- `192` 条命令全部成功，成功率 `100%`
- 总吞吐达到 `195.07 commands/s`
- 并发命令延迟均值 / P95 为 `13.34 / 25.45 ms`
- 并发 flush 均值 / P95 为 `521.26 / 598.44 ms`

这组数据的含义很直接：**Iruka_VFS 已经把“命令执行热路径”和“持久化路径”分开了。** 日常 `cat`、`search`、`edit` 主要消耗的是内存镜像和运行时索引，而不是每次都把数据库放进主路径；真正更重的部分，被收敛到了 `flush` 和后台 checkpoint。

- 命令执行均值 / P95 只有 `2.10 / 2.75 ms`
- checkpoint catch-up 均值 / P95 为 `561.37 / 741.07 ms`

这个在 Agent 调用链路里，Iruka_VFS 耗时基本可以忽略不计的。

最重的阶段已经集中在后台持久化链路，而不是前台编辑链路：

- `mirror.flush_workspace_mirror` 均值 `350.43 ms`
- `repo.bulk_insert_command_logs` 均值 `203.48 ms`
- `sqlalchemy.Session.commit`（checkpoint worker）均值 `139.68 ms`

这说明当前系统的主要工程矛盾已经不再是“命令本身慢”，而是“如何进一步压缩 checkpoint / flush 的尾延迟”。从 runtime 设计角度看，这是一个健康信号，因为它意味着前台编辑体验已经被从数据库 I/O 中解耦出来。

Iruka_VFS 的整体结论可以概括为：

> **它已经具备了“前台编辑毫秒级、后台落盘亚秒级、flush 链路经多轮 benchmark 验证可稳定成功”的 runtime 特征。**

这并不意味着系统已经没有优化空间。相反，数据已经很明确地指出了下一阶段重点：

- 继续压缩 checkpoint / flush 的长尾延迟
- 继续降低命令日志写入对后台持久化的放大效应
- 在维持 durability 的前提下，进一步提升高并发 workspace 下的吞吐上限

## 7. Iruka_VFS 适合什么，不适合什么

从实现结构上看，Iruka_VFS 更准确的定义不是：

- 一个 file tool 集合
- 一个简单虚拟目录树
- 一个轻量 shell 包装器

而是：

> **一个面向 Editing Harness 的 Workspace Runtime。**

它特别适合这类任务：

- 工作对象是一组文件
- 需要反复局部修改
- 需要保存中间状态
- 需要引用 context / skills / notes
- 需要结果最终回写宿主
- 需要失败后继续推进，而不是全部重来

例如：

- 长文编辑
- 文档重写
- 多材料归并
- 章节式创作
- 资料驱动生成
- 多阶段草稿推进

但它并不追求替代完整 sandbox。对于更偏代码执行型的 Agent，完整 shell、依赖安装、进程控制、测试运行仍然很重要。Iruka_VFS 服务的重点，是 editing-style harness，而不是所有类型的 Agent。

---

## 8. 总结

如果用一句话概括这篇文章，我会这样说：

> **Harness Agent 需要的不是越来越长的工具菜单，而是越来越好的 Runtime。对文件型任务来说，这个 Runtime 最自然的形态，就是文件工作区 Runtime。**

而 **Iruka_VFS** 的设计定位，正是这一判断的一个具体实现方向：

- 它不追求完整 OS 环境
- 它不满足于离散 file tools
- 它试图提供一层轻量、专用、可控的文件工作区 Runtime
- 它服务的是 editing-style harness，而不是所有类型的 Agent

所以，Iruka_VFS 最值得关注的，不只是它如何组织文件，而是它背后的那个判断：

> **真正的 Agent，不是“会调工具的模型”，而是“能在 Runtime 中持续推进任务、直到收敛的系统”。**

如果任务的核心对象是一组持续演进的文件，那么你迟早会发现：你需要的不是再加几个 file tools，而是一套真正的文件工作区 Runtime。
